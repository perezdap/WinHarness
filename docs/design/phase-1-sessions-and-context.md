# Phase 1 Design: Sessions, Context Engineering, and Compaction

**Status:** Implemented  
**Target:** WinHarness v0.2  
**Goal:** Pi Coder parity for session lifecycle and context engineering — persistent tree-structured sessions, full tool-aware message history, project context files, and manual compaction.

**Implementation complete:** PR-1 through PR-9 are merged. Sessions, context files, compaction, and CLI/TUI integration ship in v0.2.

This design is informed by [Pi's session format](https://github.com/earendil-works/pi/blob/main/packages/coding-agent/docs/session-format.md) but constrained by WinHarness's Native AOT, C# layering, and existing domain language in `CONTEXT.md`.

---

## 1. Problem Statement

WinHarness v0.1 has a working agent loop but three structural limits block serious multi-turn coding work:

| Limit | Impact |
|-------|--------|
| Conversations are in-memory only | Sessions lost on exit; no resume |
| Turns persist only final assistant text | Tool calls/results vanish between turns; model forgets what it read/ran |
| System prompt is hardcoded | No project-specific instructions (`AGENTS.md`) |
| No compaction | Long sessions will hit context limits with no recovery |

Phase 1 closes these gaps without tackling extensibility (Phase 4), multi-provider breadth (Phase 2), or steering queues (Phase 3).

---

## 2. Goals and Non-Goals

### Goals

1. Persist sessions as append-only JSONL files with a **tree structure** (`id` / `parentId`).
2. Store **full turn artifacts** — user, assistant (including tool calls), and tool results.
3. Load **context files** at startup: `AGENTS.md`, `CLAUDE.md`, `.winharness/SYSTEM.md`, `.winharness/APPEND_SYSTEM.md`.
4. Support **CLI resume/continue** and slash commands: `/session`, `/name`, `/tree`, `/fork`, `/compact`.
5. Remain **AOT-safe** — all session DTOs in `WinHarnessJsonSerializerContext`.

### Non-Goals (deferred)

- Auto-compaction on context overflow (manual `/compact` only).
- Session export/share (HTML, gist).
- Branch summaries generated by LLM when switching branches (`branch_summary` entries).
- `label` bookmarks, `custom` extension entries, thinking-level tracking.
- RPC/JSON print modes.
- Changing the Native AOT single-exe gate.

## 4. Domain Language Updates (`CONTEXT.md`)

Add these terms (keep existing ones):

**Session**  
A persisted JSONL file plus in-memory tree state. Distinct from Conversation (the ephemeral projection sent to the runtime for one turn).  
_Avoid_: Thread, chat (overloaded).

**Session entry**  
One append-only JSONL record in a session file. Forms a tree via `id` and `parentId`.  
_Avoid_: Message (reserved for ConversationMessage).

**Active branch**  
The path from the current leaf entry to the root. The Conversation for the next turn is built from this path.  
_Avoid_: Current thread.

**Turn artifacts**  
All ConversationMessages produced during one agent run: user input, assistant segments (text + tool calls), tool results. The caller appends these to the session, not just the final assistant text.

**Compaction**  
A session entry that replaces older messages in the *active context* with a summary. Full history remains in the JSONL file.

Update the **Turn** definition:

> A turn runs the agent against a Conversation built from the active branch. On completion, the caller appends all turn artifacts to the session. A turn may include multiple internal tool round-trips.

## 5. Session File Format

### 5.1 Location

```text
%APPDATA%\WinHarness\sessions\{workspaceKey}\{yyyyMMdd-HHmmss}_{sessionId}.jsonl
```

- `workspaceKey` — normalized absolute cwd with `\` → `-`, drive colon stripped, lowercased. Example: `c-users-dperez-documents-github-winharness`.
- `sessionId` — 8-character hex (first 8 chars of a UUID v4), matching Pi's short IDs.

Pi uses `~/.pi/agent/sessions/--<path>--/`; WinHarness follows the same *organize by cwd* principle but uses `%APPDATA%` for consistency with existing config/logs.

### 5.2 Version

**Version 1** — tree structure. No legacy migration needed (greenfield).

### 5.3 Line types

Each line is one JSON object. The file is append-only during normal operation; `/tree` branching only changes the in-memory leaf pointer until the next append creates new children.

#### SessionHeader (line 1, not in tree)

```json
{
  "type": "session",
  "version": 1,
  "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "timestamp": "2026-06-27T14:00:00.000Z",
  "cwd": "C:\\Users\\dperez\\Documents\\Github\\WinHarness",
  "parentSession": null
}
```

`parentSession` is set when created via `/fork` from another session file (stores absolute path to source file).

#### SessionEntryBase (all other lines)

```json
{
  "type": "<entry-type>",
  "id": "a1b2c3d4",
  "parentId": "prev1234",
  "timestamp": "2026-06-27T14:00:01.000Z",
  ...
}
```

- `id` — 8-char hex, unique within file.
- `parentId` — `null` only for the first tree entry.
- `timestamp` — ISO 8601 UTC.

#### Entry types (Phase 1)

| `type` | Purpose | In LLM context? |
|--------|---------|-----------------|
| `message` | User, assistant, tool messages | Yes |
| `compaction` | Context summary boundary | Yes (as system summary) |
| `model_change` | Provider/model switch | Metadata only (restores session provider/model) |
| `session_info` | Display name (`/name`) | No |

Deferred: `branch_summary`, `label`, `custom`, `thinking_level_change`.

#### `message` entry

```json
{
  "type": "message",
  "id": "b2c3d4e5",
  "parentId": "a1b2c3d4",
  "timestamp": "2026-06-27T14:00:02.000Z",
  "message": {
    "role": "assistant",
    "content": [
      { "kind": "text", "text": "I'll read the file." },
      { "kind": "toolCall", "id": "call_01", "name": "read_file", "argumentsJson": "{\"path\":\"README.md\"}" }
    ],
    "providerId": "openai-main",
    "modelId": "gpt-primary",
    "usage": {
      "inputTokens": 1200,
      "outputTokens": 340,
      "totalTokens": 1540
    }
  }
}
```

`usage` is optional on assistant messages (decision #2). Populate from provider `UsageDetails` when available; omit when the provider does not report tokens.

Tool result messages store **full output text verbatim** (decision #5), including large `glob`/`grep` results. Compaction is the mechanism for managing context size, not truncation at write time.

#### `compaction` entry

```json
{
  "type": "compaction",
  "id": "f6g7h8i9",
  "parentId": "e5f6g7h8",
  "timestamp": "2026-06-27T14:10:00.000Z",
  "summary": "User asked to add session persistence. Implemented JSONL store and ...",
  "firstKeptEntryId": "c3d4e5f6",
  "tokensBefore": 42000
}
```

#### `model_change` entry

```json
{
  "type": "model_change",
  "id": "d4e5f6g7",
  "parentId": "c3d4e5f6",
  "timestamp": "2026-06-27T14:05:00.000Z",
  "providerId": "local-ollama",
  "modelId": "local-coder"
}
```

#### `session_info` entry

```json
{
  "type": "session_info",
  "id": "k1l2m3n4",
  "parentId": "j0k1l2m3",
  "timestamp": "2026-06-27T14:35:00.000Z",
  "name": "Phase 1 sessions design"
}
```

### 5.4 Tree example

```text
[session header]

[user] ─── [assistant+tools] ─── [tool results] ─── [assistant] ─── [user] ─┬─ [user]  ← leaf (current)
                                                                              │
                                                                              └─ [user]  ← branch
```

Branching (`/tree`) moves the leaf to an earlier entry. The next user message appends as a **child** of that entry, creating a sibling branch. All branches remain in one file.

## 6. Message Model Changes

### 6.1 Replace flat `ConversationMessage`

Current:

```csharp
public sealed record ConversationMessage(ConversationRole Role, string Content);
```

Proposed:

```csharp
public sealed record ConversationMessage(
    ConversationRole Role,
    IReadOnlyList<ContentBlock> Content,
    string? ProviderId = null,
    string? ModelId = null);

public abstract record ContentBlock;

public sealed record TextBlock(string Text) : ContentBlock;

public sealed record ToolCallBlock(
    string Id,
    string Name,
    string ArgumentsJson) : ContentBlock;

public sealed record ToolResultBlock(
    string ToolCallId,
    string ToolName,
    string Text,
    bool IsError = false) : ContentBlock;
```

**Migration within v0.2:** Add factory helpers so existing call sites can use `ConversationMessage.FromText(role, text)` during the transition. JSON serialization uses a discriminated `kind` field on content blocks (AOT-safe polymorphism via `JsonPolymorphic`).

### 6.2 Conversation role mapping to `ChatMessage`

`SessionContextBuilder` and `SingleAgentRuntime.ProjectConversation` map blocks to `Microsoft.Extensions.AI` types:

| Role | MEAI mapping |
|------|--------------|
| User | `ChatMessage(ChatRole.User, textBlocks)` |
| Assistant | `ChatMessage` with `TextContent` + `FunctionCallContent` |
| Tool | `ChatMessage(ChatRole.Tool, FunctionResultContent)` per tool result block |
| System | `ChatMessage(ChatRole.System, text)` — compaction summaries, skills, context files |

### 6.3 Turn completion contract

Replace the terminal `Completed` event payload:

**Before:**

```csharp
public sealed record AgentEvent(..., ConversationMessage? AssistantMessage = null);
```

**After:**

```csharp
public sealed record TurnArtifacts(IReadOnlyList<ConversationMessage> Messages);

public sealed record AgentEvent(
    AgentEventKind Kind,
    string Message,
    TurnArtifacts? TurnArtifacts = null);
```

`TurnArtifacts.Messages` contains, in order:

1. The user message for this turn (echo of input — lets session store stay self-contained).
2. Zero or more assistant messages (each may include tool calls).
3. Zero or more tool result messages.
4. Final assistant message (text only, if the model ended with text).

The CLI appends each message as a separate `message` session entry (chained via `parentId`), not as one blob.

## 7. Turn Recording (Runtime)

### 7.1 Problem

`UseFunctionInvocation()` handles tool loops inside `IChatClient`. The current runtime only concatenates `ChatResponseUpdate.ToString()` text deltas and loses tool call structure.

### 7.2 Solution: `TurnRecorderChatClient`

Wrap the inner `IChatClient` returned by `IChatProvider.CreateChatClient()`:

```csharp
internal sealed class TurnRecorderChatClient : IChatClient
{
    private readonly IChatClient _inner;
    private readonly List<ConversationMessage> _recorded = [];

    public IReadOnlyList<ConversationMessage> RecordedMessages => _recorded;

    // Delegate GetResponseAsync / GetStreamingResponseAsync to _inner,
    // but also observe ChatResponse.Messages on completion.
}
```

Implementation approach:

1. Prefer `GetResponseAsync` internally after streaming completes (if the middleware exposes final `ChatResponse`) **or**
2. Accumulate `ChatResponseUpdate` contents and reconstruct assistant/tool messages using the same rules as MEAI's response builder.

`SingleAgentRuntime.RunAsync` wraps the client:

```csharp
TurnRecorderChatClient recorder = new(client);
using IChatClient chatClient = recorder.WithFunctionInvocation();
// ... stream as today ...
yield return new AgentEvent(AgentEventKind.Completed, "completed", recorder.BuildTurnArtifacts(userMessage));
```

Add unit tests mirroring `SingleAgentRuntimeTests.PassesToolsToChatClient` that assert `TurnArtifacts` contains tool call and tool result messages.

## 8. Session Manager API

### 8.1 `ISessionStore` (persistence primitive)

```csharp
public interface ISessionStore
{
    ValueTask<SessionFile> CreateAsync(string cwd, CancellationToken cancellationToken);
    ValueTask<SessionFile> OpenAsync(string path, CancellationToken cancellationToken);
    ValueTask AppendAsync(string path, SessionEntry entry, CancellationToken cancellationToken);
    ValueTask<IReadOnlyList<SessionSummary>> ListAsync(string cwd, CancellationToken cancellationToken);
}
```

`SessionFile` holds the header plus parsed entries (loaded entirely into memory — acceptable for Phase 1; typical sessions are < 10 MB).

Append uses `FileStream` with `FileMode.Append, FileAccess.Write, FileShare.Read` and writes one JSON line per call. No in-place edits.

### 8.2 `ISessionManager` (tree + active branch)

```csharp
public interface ISessionManager
{
    bool IsPersisted { get; }
    string? SessionFilePath { get; }
    string? LeafEntryId { get; }

    SessionHeader Header { get; }
    string? DisplayName { get; }

    // Static factories
    static ISessionManager InMemory(string cwd);
    static Task<ISessionManager> CreateAsync(string cwd, CancellationToken ct);
    static Task<ISessionManager> OpenAsync(string path, CancellationToken ct);
    static Task<ISessionManager> ContinueRecentAsync(string cwd, CancellationToken ct);
    static Task<IReadOnlyList<SessionSummary>> ListAsync(string cwd, CancellationToken ct);

    // Tree navigation
    void BranchTo(string entryId);
    IReadOnlyList<SessionEntry> GetActiveBranch();
    IReadOnlyList<SessionEntry> GetChildren(string entryId);
    SessionTree GetTree();

    // Mutation (returns new entry id)
    ValueTask<string> AppendMessagesAsync(
        IReadOnlyList<ConversationMessage> messages,
        CancellationToken cancellationToken);

    ValueTask<string> AppendModelChangeAsync(
        string providerId, string modelId, CancellationToken cancellationToken);

    ValueTask<string> AppendCompactionAsync(
        string summary, string firstKeptEntryId, long? tokensBefore,
        CancellationToken cancellationToken);

    ValueTask<string> AppendSessionInfoAsync(string name, CancellationToken cancellationToken);

    // Context projection
    Conversation BuildConversation(IContextFileLoader context, SkillDefinition? skill);
}
```

`BuildConversation` delegates to `SessionContextBuilder`:

1. Walk active branch root → leaf.
2. Find the latest `compaction` entry on the path. If present:
   - Emit a system message with the summary.
   - Include messages from `firstKeptEntryId` forward (skip earlier messages on the path).
3. Convert `message` entries to `ConversationMessage` list.
4. Prepend (in order): runtime system prompt, `SYSTEM.md` replacement/append, concatenated `AGENTS.md` content, selected skill system prompt.
5. Ensure the result ends with a user message before returning (caller adds the new user message for the turn, or the last entry on the branch is already a user message awaiting response — see §9).

### 8.3 `SessionSummary` (for `--resume`)

```csharp
public sealed record SessionSummary(
    string FilePath,
    string SessionId,
    string? DisplayName,
    string? FirstUserPreview,
    DateTimeOffset LastModified,
    int MessageCount);
```

## 9. Context File Loading

### 9.1 `IContextFileLoader`

```csharp
public sealed record ProjectContext(
    string? SystemPromptReplacement,
    string? SystemPromptAppend,
    string AgentsInstructions);

public interface IContextFileLoader
{
    ProjectContext Load(string workspaceRoot);
}
```

### 9.2 Discovery order

**`AGENTS.md` / `CLAUDE.md`** — walk from workspace root **down** to cwd (root first, cwd last; later files append):

| Scope | Path |
|-------|------|
| Global | `%APPDATA%\WinHarness\AGENTS.md` |
| Ancestors | `{ancestor}\AGENTS.md`, `{ancestor}\CLAUDE.md` for each parent up to drive root |
| Workspace | `{cwd}\AGENTS.md`, `{cwd}\CLAUDE.md` |

Both filenames are supported; both contribute if present.

**`SYSTEM.md`** — first match wins (replacement):

1. `{cwd}\.winharness\SYSTEM.md`
2. `%APPDATA%\WinHarness\SYSTEM.md`

**`APPEND_SYSTEM.md`** — first match wins (append):

1. `{cwd}\.winharness\APPEND_SYSTEM.md`
2. `%APPDATA%\WinHarness\APPEND_SYSTEM.md`

### 9.3 Runtime injection order

`SingleAgentRuntime` receives `ProjectContext` via `AgentRunRequest` (new field) or a scoped `IContextFileLoader` injected at runtime construction. Final system messages, in order:

1. `SYSTEM.md` replacement **or** built-in `RuntimeSystemPrompt`
2. `APPEND_SYSTEM.md` (if present)
3. `AGENTS.md` concatenation (if non-empty)
4. Selected skill prompt (transient per turn, not stored in session — same as today)

Skills remain session-selected and transient; they are **not** written to the session file.

### 9.4 Startup banner

REPL/TUI banner adds a line when context files are loaded:

```text
context: AGENTS.md (3 files, 4.2 KB) · SYSTEM.md (project)
```

## 10. Compaction

### 10.1 Manual `/compact [instructions]`

Triggered from slash command. Flow:

1. `SessionContextBuilder` builds the current active-branch conversation (without a new user message).
2. If fewer than 4 messages, refuse ("nothing to compact").
3. Run a **summarization turn** using the session's active provider/model (decision #4; `--compact-model` deferred):
   - System: "Summarize the conversation for continuation. Preserve file paths, decisions, and unfinished work."
   - User: optional custom instructions from `/compact ...` argument.
   - Messages: active branch messages (post-compaction overlay).
4. Append `compaction` entry with:
   - `summary` = summarization output
   - `firstKeptEntryId` = id of the last 2 user-assistant pairs on the branch (configurable constant: `CompactionKeepEntries = 4` messages)
   - `tokensBefore` = from last provider usage diagnostic if available, else `null`
5. Print confirmation with approximate character reduction.

Full history remains in the JSONL file. `/tree` can still navigate to pre-compaction entries.

### 10.2 Context rebuild with compaction

When building conversation for the next turn, if a `compaction` entry exists on the active branch:

```text
[System] <compaction summary>
... messages from firstKeptEntryId to leaf ...
```

## 11. CLI and Slash Command Changes

### 11.1 `winharness chat` flags

| Flag | Short | Behavior |
|------|-------|----------|
| `--continue` | `-c` | Open most recent session for cwd, or create new if none |
| `--resume` | `-r` | Interactive session picker (Spectre selection table) |
| `--session <path\|id>` | | Open specific file or match by session id suffix |
| `--no-session` | | In-memory session (today's behavior) |
| `--name <name>` | `-n` | Set display name on new session (writes `session_info` after header) |

Default when no flags: **`--continue`** (decided; breaking change from v0.1 ephemeral default — document in README). Use `--no-session` for one-off ephemeral chats.

One-shot `winharness chat --prompt "..."` without `--continue` remains ephemeral unless `--session` is passed.

### 11.2 New slash commands

| Command | Behavior |
|---------|----------|
| `/session` | Show file path, id, display name, leaf id, message count, provider/model |
| `/name <name>` | Append `session_info` entry |
| `/tree` | Navigate branch (TUI: tree view; REPL: numbered list + pick) |
| `/fork` | Create new session file in the **current workspace session dir** (new timestamped filename), branched from the active path; opens in new manager |
| `/compact [instructions]` | Manual compaction (§10) |
| `/new` | **Changed:** creates new session file (not just clear in-memory). Old file remains on disk. |
| `/resume` | In-session picker (same as `--resume`) |

Existing commands unchanged: `/provider`, `/model`, `/skills`, `/skill`, `/markdown`, `/exit`.

`/model` and `/provider` also append `model_change` entries when persistence is enabled.

## 12. Persistence and Concurrency

- **Auto-save:** every append flushes to disk immediately (same durability model as `ConfigStore.SaveAsync`).
- **Single-writer assumption:** one WinHarness process per session file. No file locking in Phase 1.
- **Corrupt line handling:** on load, skip malformed lines with a diagnostic warning; fail if header is missing or invalid.
- **Max file size:** no hard limit in Phase 1; log a warning when file exceeds 50 MB.

## 13. TUI: `/tree` View

`ChatTuiApp` adds a modal `TreeView` (Terminal.Gui `TreeView` or `ListView` with indentation):

- Shows active branch highlighted.
- `Enter` on a node calls `sessionManager.BranchTo(entryId)`.
- `Esc` closes without branching.
- After branch, scrollback rebuilds from the new active branch.

REPL fallback: print numbered entries from `GetActiveBranch()` ancestors plus children at leaf, prompt `branch-to #`.

## 17. Design Decisions

| # | Question | Decision |
|---|----------|----------|
| 1 | Default to `--continue` or stay ephemeral? | **`--continue`** (Pi parity). Opt out via `--no-session` or `sessions.defaultMode: "ephemeral"`. |
| 2 | Store provider usage on each assistant message? | **Yes** — optional `usage` object on assistant `message` entries (enables future token/cost footer). |
| 3 | `/fork` — new file in same workspace dir or prompt for location? | **Same workspace dir** — new timestamped filename under the current cwd's session folder. |
| 4 | Compaction summarization model — always active model? | **Active provider/model in Phase 1.** Defer `--compact-model` / config override to a later release. |
| 5 | Include `glob`/`grep` tool results verbatim in session? | **Yes** — full tool result text stored verbatim (Pi parity). |

## 18. Success Criteria

Phase 1 is complete when:

1. `winharness chat -c` resumes a prior session with tool history intact across turns.
2. `AGENTS.md` in the repo root appears in the model's context on startup.
3. `/tree` can branch to an earlier message and continue with a different prompt.
4. `/compact` reduces active context while the JSONL file retains full history.
5. Native AOT publish gate passes with zero warnings.
6. All new session DTOs are source-generated and covered by unit tests.
