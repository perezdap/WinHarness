# ADR-0003: Limit v0.1 providers to OpenAI-compatible endpoints

## Status

Accepted

## Context

The original target included Anthropic and OpenAI-compatible providers. Native AOT is the hardest constraint in WinHarness, and the official Anthropic SDK currently appears to be the highest-risk dependency because upstream AOT support is unresolved.

The user explicitly approved narrowing v0.1 to OpenAI-compatible endpoints.

## Decision

WinHarness v0.1 supports only OpenAI-compatible endpoints.

In scope:

- OpenAI
- OpenAI-compatible gateways and routers
- Ollama `/v1`
- LM Studio `/v1`
- vLLM `/v1`
- OpenRouter-compatible endpoints
- Azure-style OpenAI-compatible `/openai/v1` endpoints when usable through the official SDK

Out of scope for v0.1:

- Anthropic-native API support
- Provider-specific SDK integrations other than the official OpenAI SDK

## Rationale

This scope protects the Native AOT release goal by removing a known high-risk dependency from the v0.1 dependency graph. It also keeps the first release useful because many local and hosted systems expose OpenAI-compatible APIs.

The architecture remains provider-neutral through `IChatProvider`, `IProviderFactory`, and model-specific `ProviderCapabilities`, so future non-OpenAI providers can be added without rewriting the runtime.

## Consequences

- The `Anthropic` NuGet package is not referenced in v0.1.
- Anthropic is documented in `ROADMAP.md`.
- Phase 0 validates only the OpenAI-compatible provider path.
- Provider capability metadata remains model-specific and endpoint-configurable.
